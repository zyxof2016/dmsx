# DMSX PR / Code Review 检查清单

> 目的：让 `DMSX` 后续每个 PR 都按照统一质量标准审查，避免“能跑但不可维护”“功能完成但体验差”“暂时可用但后续难治理”的问题。

## 1. 使用方式

提交者在发起 PR 前自查一遍，评审者在 Review 时至少覆盖：

1. 工程边界
2. 正确性与失败路径
3. 安全与权限
4. 可观测与可运维性
5. 测试与文档
6. 前端体验与易用性

原则：

- 不只看“有没有实现”，还要看“是否容易长期维护”。
- 不只看 happy path，还要看错误路径、超时、并发、回滚、降级。
- 不只看代码风格，还要看职责边界、命名准确性和用户体验。

## 2. 提交者自查

PR 提交前必须确认：

- 改动属于正确的 `crate`、模块和目录，没有把逻辑临时堆到入口文件。
- 新增接口、字段、环境变量、配置项已经更新文档。
- 高风险操作已经补上权限、确认或审计逻辑。
- 所有用户可见改动都补了加载态、错误态、空态或成功反馈。
- 本地至少完成一次相关编译、测试或最小验证。

## 3. 工程结构检查

- 是否遵守 `docs/ENGINEERING_STANDARDS.md` 中的模块职责要求。
- `main.rs`、handler、组件入口是否仍然保持薄层。
- 是否引入了新的“大文件”“大函数”“万能工具模块”。
- 是否存在本应下沉到 `service` / `repo` / `adapter` 的逻辑被放错层。
- 是否新增了跨层耦合或反向依赖。
- 公共能力是否抽象在合适位置，而不是复制粘贴。

## 4. Rust 代码质量检查

- 类型设计是否足够明确，是否避免了魔法字符串和裸 `Uuid` 滥传。
- 错误处理是否保留了足够上下文。
- 是否存在不必要的 `unwrap()` / `expect()`。
- 异步代码是否有超时、取消、重试或退出路径。
- 是否误把 IO 密集和 CPU 密集工作放在同一个执行模型中。
- 命名是否表达真实业务语义，而不是实现细节。
- 是否有复杂逻辑缺少必要注释。

## 5. API / 契约检查

- HTTP 接口是否遵守现有路径、状态码、错误响应规范。
- 输入 DTO 是否完成校验。
- 输出是否泄漏了内部字段或敏感信息。
- Proto / WebSocket（如使用）/ LiveKit Data Channel 载荷 / 前后端 DTO 变更是否兼容。
- 列表接口是否保持分页、筛选、排序能力。
- 写接口是否定义了幂等性、审计行为和失败语义。

## 6. 数据库与状态流检查

- SQL 是否放在合适层，是否避免 `SELECT *`。
- 事务边界是否清晰。
- 并发写入是否考虑乐观锁、版本号、幂等键或去重机制。
- 状态流转是否完整，是否覆盖非法状态输入。
- migration 是否与代码同步提交。
- 数据模型、领域模型、DTO 是否被错误混用。

## 7. 安全检查

- 是否校验租户与权限：路径 `{tenant_id}` 与 JWT 许可集合、**有效角色**（`tenant_roles` / `roles`）、资源归属及写操作边界（见 [`API.md`](API.md)）。
- 是否有新增敏感操作却缺少确认或审批钩子。
- 日志、错误、审计里是否泄漏凭据、token、证书、敏感脚本内容。
- 外部调用是否有允许列表、超时和最小权限约束。
- 开发环境的“临时放行”是否被错误带入生产路径。

## 8. 可观测性与运维检查

- 关键流程是否带必要日志字段，如 `tenant_id`、`device_id`、`command_id`、`session_id`。
- 是否为关键新能力补了指标或至少保留了后续接指标的位置。
- 失败时是否能够从日志快速定位请求、设备、会话、命令。
- 长连接、后台任务、轮询流程是否可停止、可恢复、可诊断。
- 配置项是否集中解析并且命名清晰。

## 9. 测试检查

- 是否补了对高风险逻辑真正有价值的测试。
- 测试是否验证行为和边界，而不是复述实现。
- bug 修复是否带回归测试。
- mock / fixture / 时间 / 随机值是否可控。
- 如果本次未加测试，PR 描述里是否说明原因和替代验证方式。

## 10. 前端与体验检查

- 是否有加载态、空态、错误态、成功反馈。
- 表单是否有默认值、占位提示、即时校验。
- 危险操作是否有二次确认且视觉上足够醒目。
- 文案是否面向用户、可理解、可指导下一步。
- 页面是否保持与现有设计 token、颜色语义、组件风格一致。
- 批量任务、长耗时任务、失败重试是否足够易用。
- AI 结果、策略建议、远控执行结果是否具备可解释性。

## 11. 常见直接拦截项

以下情况原则上不应通过 Review：

- 新逻辑继续堆进超大 `main.rs` / 超大组件 / 超大 handler
- 使用 `unwrap()` 掩盖真实失败路径
- 没有权限校验就开放写操作
- 高风险操作没有审计
- 接口字段已变更但前后端/Proto/文档未同步
- 只实现 happy path，没有错误反馈
- 引入临时脚本式写法但没有回收计划

## 12. PR 描述推荐模板

建议每个 PR 描述至少包含：

### 变更目标

- 解决了什么问题
- 为什么现在要做

### 主要改动

- 模块 / 接口 / 数据结构层面的关键变化

### 风险点

- 哪些地方最容易出问题
- 做了什么保护

### 验证方式

- 编译
- 测试
- 手工验证

### 文档更新

- 更新了哪些文档

## 13. 评审结论标准

只有当以下问题都得到正面回答，PR 才应视为可合并：

1. 这次改动是否让系统更清晰，而不是更混乱？
2. 这次改动是否让后续继续开发更容易，而不是更难？
3. 这次改动在失败、超时、异常输入下是否仍然可控？
4. 这次改动是否兼顾了安全、可观测、测试和文档？
5. 这次改动对最终使用者是否更友好、更易懂、更顺手？
