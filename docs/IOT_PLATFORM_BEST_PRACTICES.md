# DMSX IoT 平台信息架构与最佳实践对齐

本文档定义 DMSX 管理台后续演进的产品信息架构基线。当前结论：DMSX 已具备多租户、设备注册、命令、策略、制品、审计、远程桌面的主链路，但管理台应继续按主流 IoT 平台的职责边界收敛，避免把平台治理、租户运营、设备生命周期和权限维护混在同一页面。

## 参考对象

- AWS IoT Core：Fleet indexing 用于按设备注册表、影子、连接状态、违规状态等搜索与聚合设备；IoT Jobs 用于向一个或多个设备下发远程操作、固件/应用更新、重启、证书轮换等任务，并支持 rollout、abort、timeout、retry 等配置。
- Azure IoT Hub：Device twins 承载设备元数据、期望状态、上报状态，并用于查询设备；Automatic device management 通过 twin tags/reported properties 选择目标设备，滚动应用配置并观察系统/自定义指标。
- ThingsBoard：系统管理员负责租户与租户管理员；租户管理员维护设备、资产、客户、仪表盘、规则引擎等租户内对象；Tenant Profile / Device Profile 用于统一限额、速率、传输和处理配置。

这些平台的共性不是菜单名字相同，而是把 **平台面、租户面、设备生命周期、权限与配额、批量运维、观测与审计** 分成清晰边界。

## 架构原则

1. 登录后先选择工作范围：用户必须先完成认证，再基于账号拥有的平台权限和租户权限进入“平台管理”或某个“租户管理”。平台角色不穿透到租户路径，租户角色也不参与平台治理。
2. 平台面只做全局治理：租户目录、租户配额、平台权限、平台审计、平台健康、全局配置属于平台面；不在平台首页直接混放设备、策略、命令的租户级维护表单。
3. 租户面围绕设备生命周期组织：设备接入、设备分组、设备影子、配置/策略、命令/作业、应用分发、合规、安全、审计、观测都属于租户面。
4. 权限按对象拆页：用户、角色、菜单、策略分别维护；总览页只做摘要和入口。对象页采用“单列表 + 抽屉”承载详情或轻量编辑，避免一个权限页塞满所有表单。
5. 配额和配置需要有归属：平台配额维护租户级上限和全局默认值；后续租户侧可维护租户内 profile / device profile，但不能与平台配额混在同一维护入口。
6. 批量运维要从命令升级为作业：单设备命令适合即时排障；批量升级、批量脚本、证书轮换、重启、远程诊断应演进为带目标选择、灰度、暂停、失败阈值、重试和状态聚合的 Job。
7. 设备查询要从列表升级为 Fleet 视角：设备列表应继续补齐分组、标签、平台、在线状态、影子字段、版本、合规状态等筛选和聚合，支撑“找一批设备并操作它们”。

## 目标信息架构

### 平台管理

| 模块 | 页面 | 职责 |
|------|------|------|
| 平台工作台 | 平台总览 | 平台级计数、风险摘要、模块入口，不直接承载对象维护 |
| 权限中心 | 权限总览 / 角色 / 用户 / 菜单 / 权限策略 | 平台权限对象分离维护，单列表 + 抽屉模式 |
| 运营治理 | 租户管理 / 配额治理 | 跨租户目录、租户创建、租户级资源上限和使用率 |
| 平台运维 | 全局审计 / 平台健康 / 系统设置 | 跨租户排障、组件健康、全局配置 |

### 租户管理

| 模块 | 页面 | 职责 |
|------|------|------|
| 租户工作台 | 态势总览 / AI 智慧中心 | 租户内设备、命令、合规、趋势与智能分析 |
| 设备运营 | 设备管理 / 远程命令 | 设备注册、批量预注册、详情、影子、即时命令、远程桌面 |
| 策略安全 | 策略中心 / 合规 / 策略编辑器 / 审计 / 用户角色 | 策略、合规发现、租户 RBAC、租户审计 |
| 交付配置 | 应用分发 / 网络管控 | Agent 制品、安装/升级元数据、网络与站点能力 |

## 当前对齐状态

- 已完成：登录后选择平台/租户范围；平台与租户权限分离；平台首页独立；平台权限拆成角色、用户、菜单、权限策略页面；左侧菜单按模块分组；平台租户目录、配额、全局审计、平台健康已有入口。
- 部分完成：设备注册和零接触安装已可用，但 Fleet 分组、动态查询、批量作业模型尚未完整产品化；`install_update` 已可下发，但仍是命令增强，不是完整 OTA / Job 系统。
- 待补齐：平台用户目录和平台角色绑定的后端 CRUD、菜单权限映射的持久化、租户 profile / device profile、设备组/标签的主界面、批量 Job、灰度发布、失败阈值与回滚策略、跨设备观测聚合。

## 下一步落地顺序

1. 稳定信息架构：保持平台/租户两套工作区；左侧菜单按模块分组；平台首页只做摘要和入口。
2. 补齐平台权限后端：新增平台用户目录、平台角色绑定、菜单权限映射和权限策略持久化 API，让当前权限中心从结构化页面升级为真实维护入口。
3. 做 Fleet 管理基础：在设备侧补设备组/标签/profile 的列表、筛选和批量选择能力，为后续 Job 打基础。
4. 建立 Job / OTA 模型：把批量升级、批量脚本、重启、证书轮换等从普通 command 抽象为可跟踪的 Job，包含目标、rollout、abort、retry、timeout 和执行聚合。
5. 强化观测与审计：按平台、租户、设备组、Job 聚合健康、合规、命令结果和审计事件。
