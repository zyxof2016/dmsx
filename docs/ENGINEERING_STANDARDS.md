# DMSX 工程规范与开发约定

> 目标：基于当前 `DMSX` 的 Rust workspace 现状，建立一套后续开发必须遵守的工程规范，确保系统**健壮可维护、模块清晰、功能完整、操作易用、界面美观**。
>
> 本规范适用于：
> - `crates/dmsx-core`
> - `crates/dmsx-ai`
> - `crates/dmsx-api`
> - `crates/dmsx-device-gw`
> - `crates/dmsx-agent`
> - `web/`

## 1. 总体原则

### 1.1 核心目标

后续所有开发必须同时满足以下 6 个目标：

1. **正确性优先**：先保证行为正确、边界清晰、失败可控，再追求“快写完”。
2. **可维护优先**：代码必须容易读、容易改、容易定位问题，禁止用短期技巧换长期成本。
3. **模块化优先**：按职责拆分 crate、模块、服务、组件，禁止“大文件 + 大函数 + 大状态对象”持续膨胀。
4. **接口稳定优先**：内部模块边界、外部 API、Proto、前端 DTO 一旦发布，变更必须可追踪、可兼容、可迁移。
5. **安全与可观测内建**：鉴权、审计、日志、追踪、限流、幂等不是附加项，而是默认要求。
6. **产品体验优先**：功能不只“能用”，还必须“好用、易懂、反馈及时、默认安全”。

### 1.2 强制约束

以下规则为强制项：

- 所有新增能力必须先明确归属到某个 `crate`、模块和文档位置。
- 所有写路径必须定义错误语义、日志字段、审计行为和超时策略。
- 所有外部接口必须有输入校验、状态码规范和可定位的错误信息。
- 所有长生命周期任务必须支持取消、超时、重试退避和优雅关闭。
- 所有高风险操作必须具备确认机制、权限约束和审计记录。
- 所有 UI 交互必须有加载态、空态、错误态和成功反馈。

## 2. Workspace 与 crate 职责

### 2.1 crate 边界

当前项目应持续保持以下职责边界：

- `dmsx-core`
  - 只放**领域模型、公共 DTO、统一错误、跨 crate 共用类型**。
  - 禁止放网络请求、数据库连接、文件系统、线程管理等基础设施逻辑。
- `dmsx-ai`
  - 只放 AI 能力抽象、策略推荐、异常检测、预测逻辑、LLM 适配层。
  - 不直接承担 HTTP 路由和数据库连接池装配。
- `dmsx-api`
  - 负责 REST 接口、认证鉴权、中间件、应用服务编排（按需可扩展 WebSocket 等长连接，**远程桌面主链路不经由 API 中继视频**）。
  - 不承载纯领域定义，不直接塞入大量业务常量和临时 JSON 拼接逻辑。
- `dmsx-device-gw`
  - 负责设备接入、gRPC 流、会话管理、背压、mTLS、网关级限流。
  - 不应演化为“第二个控制面”。
- `dmsx-agent`
  - 负责设备侧采集、命令执行、远控、远程桌面、上报。
  - 平台差异逻辑必须封装，不允许散落在全局代码中。
- `web/`
  - 负责展示、交互、状态管理和前端侧体验。
  - 不在组件内部写散乱 API 访问、领域转换和权限判断。

### 2.2 依赖方向

必须遵守单向依赖：

`dmsx-core` <- `dmsx-ai` / `dmsx-api` / `dmsx-device-gw` / `dmsx-agent`

禁止：

- `dmsx-core` 反向依赖上层 crate
- `dmsx-api` 依赖 `dmsx-agent`
- 前端类型与后端 DTO 长期脱节

如出现共用逻辑，应优先下沉到：

- `dmsx-core`：纯类型、校验、错误、领域约束
- 新增独立 crate：当逻辑明显属于“基础设施公共层”时再拆

## 3. 模块化标准

### 3.1 单文件规模约束

强制约束：

- 单个 Rust 源文件建议不超过 `400` 行。
- 超过 `600` 行必须拆分。
- 单个函数建议不超过 `80` 行。
- 超过 `120` 行的函数必须重构。

结合当前项目：

- `dmsx-agent/src/main.rs` 已经承担过多职责，后续新增功能**不得继续堆积在同一文件**。
- 新功能必须按模块拆分，并逐步把现有能力迁移出去。

### 3.2 推荐目录结构

#### `dmsx-api`

```text
src/
├── main.rs              # 只做启动与装配
├── config.rs            # 环境变量与配置解析
├── app.rs               # Router / middleware / state 装配
├── auth/                # 鉴权、中间件、claims、权限检查
├── routes/              # 路由聚合
├── handlers/            # HTTP handler，薄层
├── services/            # 应用服务，编排事务与业务流程
├── repos/               # sqlx 查询与持久化
├── dto/                 # 请求响应对象
├── desktop/             # 远程桌面领域
└── observability/       # tracing/metrics/otel
```

#### `dmsx-agent`

```text
src/
├── main.rs              # 只保留启动入口
├── app.rs               # 启动流程、任务编排、优雅退出
├── config.rs            # AgentConfig
├── device.rs            # 注册、身份、设备元数据
├── telemetry.rs         # 遥测采集
├── command_runner.rs    # 命令拉取、执行、结果提交
├── desktop.rs           # 远程桌面采集与输入处理
├── rustdesk.rs          # RustDesk 集成
├── platform/
│   ├── mod.rs
│   ├── linux.rs
│   ├── windows.rs
│   └── android.rs
└── utils.rs
```

### 3.3 模块职责规则

- `handler` 只负责：参数提取、调用 service、映射 HTTP 响应。
- `service` 只负责：业务编排、事务边界、权限检查、调用 repo/adapter。
- `repo` 只负责：数据库读写，不夹带业务流程判断。
- `domain` 只负责：领域类型、约束、状态转换、校验。
- `adapter` 只负责：外部系统对接，如 LLM、S3、NATS、LiveKit。

禁止：

- 在 handler 中直接写复杂 SQL
- 在 repo 中直接做权限判定
- 在 DTO 中混入数据库模型
- 在 `main.rs` 中堆业务逻辑

## 4. Rust 代码规范

### 4.1 类型设计

- 优先使用强类型，而不是字符串和裸 `Uuid` 到处传递。
- 保持当前 `dmsx-core` 的 newtype 方向，继续为关键标识定义透明新类型。
- 枚举必须使用穷举语义表达状态，禁止魔法字符串。
- 复杂 JSON 字段必须逐步演进为结构化类型，而不是长期停留在 `serde_json::Value`。

推荐：

- 外部边界可接受 JSON
- 进入业务层后尽快转为显式结构体

### 4.2 `pub` 暴露控制

- 默认私有，按需暴露。
- 优先 `pub(crate)`，谨慎使用 `pub`。
- `lib.rs` 应作为对外导出面，避免调用方深入依赖内部模块结构。

### 4.3 函数设计

每个函数只做一件事，并满足：

- 输入明确
- 副作用明确
- 错误明确
- 日志明确

禁止：

- 一个函数同时做“解析 + 校验 + 调库 + 重试 + 序列化 + 打日志”
- 返回 `bool` 表示复杂失败原因

### 4.4 错误处理

统一规范：

- 领域与库层使用 `thiserror`
- 进程边界或命令行入口可使用 `anyhow`
- HTTP 层统一映射到 RFC 7807 Problem Details
- 错误消息必须可定位，包含关键业务上下文

结合当前项目：

- 继续以 `dmsx-core::DmsxError` 作为公共错误契约
- 不允许大量 `map_err(|_| DmsxError::Internal("xxx".into()))` 丢失上下文
- 对外返回的信息要安全，对内日志要充分

推荐格式：

- 对用户：简洁、可理解、不泄漏敏感信息
- 对日志：带 `tenant_id`、`device_id`、`command_id`、操作名、失败原因

### 4.5 `unwrap` / `expect` 规则

- 业务代码禁止滥用 `unwrap()`。
- `expect()` 只允许用于“进程启动即失败”的不可恢复场景，且必须写清原因。
- 请求处理路径、后台任务、Agent 循环、网络/文件系统操作中禁止无理由 `panic`。

### 4.6 注释规范

- 优先写自解释代码，少写重复注释。
- 对复杂状态机、协议转换、并发控制、平台兼容分支，应写简洁注释。
- 公共类型、公共函数、关键 trait 必须写文档注释。

## 5. 异步、并发与长连接规范

### 5.1 Tokio 使用规则

- 明确区分 CPU 密集和 IO 密集任务。
- CPU 密集任务使用 `spawn_blocking` 或专用线程池。
- 每个后台任务都必须可追踪、可取消、可收敛。

### 5.2 任务管理

推荐：

- 使用 `JoinSet`、`tokio::select!`、取消 token 管理任务生命周期。
- 所有周期任务必须支持：
  - 初始延迟
  - 失败重试
  - 指数退避
  - 抖动
  - 关闭信号

禁止：

- 无上限 `tokio::spawn`
- 没有超时的网络调用
- 没有退出路径的死循环

### 5.3 通道与背压

- 长连接、推流、命令分发必须使用有界通道。
- 明确处理“消费者慢”“连接断开”“队列满”的策略。
- 对实时流量要优先定义丢弃策略，而不是默认无限缓存。

适用于：

- `dmsx-device-gw` 的命令流
- LiveKit 房间内 **Data Channel** 的输入事件与 Agent 侧处理（有界队列、丢弃策略）
- `dmsx-agent` 的屏幕采集帧管线与控制事件处理

## 6. 配置与环境变量规范

### 6.1 配置来源

必须统一配置读取方式：

- 启动时集中解析
- 转成强类型配置对象
- 校验后再注入应用

禁止：

- 在业务函数中零散 `std::env::var`
- 默认值逻辑散落在不同模块

### 6.2 配置分层

建议统一分为：

- 基础运行配置：监听地址、数据库、日志级别
- 外部依赖配置：Postgres、Redis、NATS、S3、LLM、LiveKit
- 安全配置：JWT、公私钥、mTLS、允许列表
- 功能开关：实验特性、灰度能力、降级策略

### 6.3 敏感信息

- 密钥、令牌、证书、DSN 不得写入源码、测试快照和普通日志。
- 示例配置必须使用占位值。
- 敏感配置必须在文档中说明来源与轮换方式。

## 7. 数据库与持久化规范

### 7.1 `sqlx` 使用标准

- 所有 SQL 必须集中在 `repo` 层。
- 优先使用显式字段列表，禁止 `SELECT *`。
- 所有写操作要明确幂等性和事务边界。
- 跨表写入必须在 service 中定义事务，不允许 repo 之间隐式串联副作用。

### 7.2 模型分离

必须分清 4 类对象：

- DB Model：数据库映射对象
- Domain Model：领域对象
- DTO：接口输入输出
- View Model：前端展示模型

禁止一个结构体同时承担这 4 类职责。

### 7.3 迁移规范

- 每次 schema 变更必须附带 migration。
- migration 必须可重复执行、可审阅、可回滚或可替代。
- 删除列、改枚举、重命名字段必须写兼容迁移方案。

### 7.4 幂等与并发控制

- 创建类接口必须支持幂等键或天然去重。
- 修改类接口必须明确乐观锁或版本字段。
- 对设备影子、命令状态、会话状态等共享资源，必须避免“最后写入覆盖一切”的隐式竞争。

## 8. API 与协议规范

### 8.1 HTTP API

- 统一使用 `/v1/...` 版本前缀。
- 错误响应统一使用 `application/problem+json`。
- 列表接口统一支持分页、筛选、排序。
- 写接口必须明确审计字段和幂等语义。

### 8.2 DTO 规范

- `dto` 独立定义，不直接复用数据库实体。
- 输入 DTO 必须先校验再进入 service。
- 输出 DTO 不暴露内部实现细节和敏感字段。

### 8.3 gRPC / Proto 规范

- Proto 是跨边界契约，字段名与枚举值变更必须兼容。
- 禁止复用保留字段号。
- 对 streaming RPC 必须定义连接建立、心跳、超时、重连与错误码语义。

### 8.4 WebSocket 规范（如使用）

- 明确消息类型、方向、序列化格式和关闭语义。
- 二进制帧与控制 JSON 必须分层定义，不允许“靠约定猜”。

### 8.5 LiveKit / 实时通道（远程桌面）

- 远程桌面以 **`session_id` 为生命周期主键**；创建、重连、显式断开、服务端回收必须可追踪。
- 键鼠等控制事件走 **Data Channel（UTF-8 JSON）**：必须定义事件 schema、版本字段、未知字段忽略策略。
- 媒体面（视频轨）与信令由 LiveKit 承担；控制面 API 只负责 token、房间名、与 Agent 命令编排，避免把视频流再绕回 `dmsx-api`。

## 9. Agent 侧专项规范

### 9.1 平台抽象

- 平台差异逻辑必须封装到 `platform/*` 模块。
- Android、Linux、Windows 分支不得长期通过 `if cfg!(...)` 穿插在大函数里。
- 所有平台特性必须定义能力探测，而不是默认假设“平台一定支持”。

### 9.2 命令执行

- 命令执行必须有超时、退出码、stdout/stderr 截断策略。
- 危险命令必须有 allowlist、审批或显式风险标识。
- 命令结果上报必须区分：
  - 网络失败
  - 执行失败
  - 结果过大
  - 超时取消

### 9.3 遥测采集

- 遥测字段必须稳定、可版本化、可兼容。
- 高成本采集项必须可关闭或降频。
- 采集失败不得拖垮主循环。

### 9.4 远程桌面

- 屏幕采集、编码、发送、输入注入必须解耦。
- 帧率、质量、带宽上限必须可配置。
- 当网络拥塞时，应优先降帧/降质量，而不是无限堆积内存。

## 10. 前端、交互与美观规范

### 10.1 页面设计原则

前端必须坚持：

- 信息分层清晰
- 关键操作突出
- 危险操作醒目
- 反馈及时可预期
- 新手可理解，熟手高效率

### 10.2 统一交互要求

所有页面必须具备：

- 加载骨架或加载态
- 空数据态
- 错误态
- 成功反馈
- 二次确认机制（高风险操作）
- 表单校验与即时提示

### 10.3 组件规范

- 页面的数据获取放在 hooks 或 page container 层，不塞进纯展示组件。
- 复杂弹窗、抽屉、表格操作区必须拆组件。
- 表格页统一具备搜索、筛选、分页、刷新、导出能力。

### 10.4 视觉规范

- 使用统一间距、圆角、色板、阴影和状态色。
- 不允许同一类信息在不同页面使用不同颜色语义。
- 危险、警告、成功、信息四类状态必须保持一致。
- 深色/浅色主题如后续支持，必须由设计 token 驱动，不允许组件内硬编码颜色。

### 10.5 易用性标准

“好用”必须可操作地落地为：

- 关键任务不超过 3 步
- 批量操作优先于重复点击
- 默认值合理
- 错误提示能指导下一步
- 复杂功能提供示例、占位提示或预填模板

适用于：

- 远控命令下发
- 策略创建与发布
- 制品上传
- AI 推荐采纳
- 远程桌面连接与断开

## 11. 安全与审计规范

### 11.1 默认安全

- 所有接口默认要求身份校验，测试/开发放行必须显式标记。
- 所有租户级资源必须校验 `tenant_id` 一致性。
- 高危操作默认不自动执行。

### 11.2 审计要求

以下操作必须写审计：

- 登录与鉴权失败
- 创建设备、策略、命令、制品
- 远控操作与远程桌面会话
- AI 建议生成、采纳、拒绝
- 权限配置变更

### 11.3 日志脱敏

不得记录：

- 密码、密钥、完整 token、证书私钥
- 敏感脚本全文
- 用户隐私数据原文

必要时只记录摘要、哈希、长度或截断内容。

## 12. 可观测性规范

### 12.1 日志

统一使用 `tracing`，日志必须结构化。

关键字段至少包含：

- `tenant_id`
- `device_id`
- `command_id`
- `policy_id`
- `session_id`
- `request_id`
- `trace_id`

### 12.2 指标

每个服务至少暴露：

- 请求量
- 错误率
- 延迟分位数
- 活跃连接数
- 队列深度
- 后台任务失败次数

### 12.3 追踪

- 跨 HTTP、gRPC、消息总线传播 trace context。
- 远程桌面、命令执行、AI 调用等长链路必须能串起来。

## 13. 测试规范

### 13.1 测试分层

必须分层测试：

- 单元测试：纯函数、校验、状态转换、解析
- 集成测试：repo、HTTP handler、Proto 契约
- 端到端测试：关键主流程

### 13.2 覆盖优先级

优先补这些测试：

- DTO 校验
- 错误映射
- 命令状态流转
- 设备影子 delta 逻辑
- AI 输出结构约束
- 远程桌面会话生命周期

### 13.3 测试质量要求

- 测试要验证行为与边界，不要只复述实现。
- 测试名称必须表达业务意图。
- 随机值、时间、外部依赖要可控。
- 关键 bug 修复必须先补回归测试。

## 14. CI/CD 与质量门禁

### 14.1 最低门禁

合并前必须通过：

- `cargo fmt --all --check`
- `cargo clippy --workspace --all-targets --all-features -D warnings`
- `cargo test --workspace`
- `cargo build --workspace`
- 前端 `pnpm lint` / `pnpm test` / `pnpm build`（如已接入）

### 14.2 审查门禁

Code Review 必查：

- 模块边界是否清晰
- 错误是否可定位
- 命名是否准确
- 新增接口是否补文档
- 并发与超时是否安全
- 是否引入了后续难以维护的“捷径”

## 15. 文档规范

以下变更必须同步更新文档：

- 新增 API / Proto / 环境变量
- 新增模块或目录结构
- 安全边界变化
- 部署方式变化
- 新的关键流程或状态机

至少更新以下对应文档之一：

- `docs/ARCHITECTURE.md`
- `docs/API.md`
- `docs/SECURITY.md`
- `docs/FRONTEND.md`
- `docs/AI_DESIGN.md`
- `docs/CHECKLIST.md`

## 16. 完成定义

一个功能只有同时满足以下条件，才算“完成”：

1. 代码已按模块落位，没有临时堆叠在入口文件。
2. 输入、输出、错误、日志、审计、权限边界已定义。
3. 至少有必要的单元测试或集成测试。
4. 文档与清单已更新。
5. 前端具备加载态、空态、错误态、成功反馈。
6. 可在本地或测试环境完成基本验证。

## 17. 当前项目的落地优先级

结合当前仓库现状，后续应优先按以下顺序治理：

1. **拆分 `dmsx-agent/src/main.rs`**
   - 先拆 `config`、`telemetry`、`rustdesk`、`command_runner`、`desktop`
2. **把 `dmsx-api` 进一步固化为 handler/service/repo 分层**
   - 避免后续业务继续堆入路由与 handler
3. **补齐测试基线**
   - 优先覆盖 `dmsx-core` 校验、`dmsx-api` 关键接口、影子与命令状态流
4. **补齐认证、RBAC、租户一致性校验**
   - 这是功能可商用前的硬门槛
5. **建立前端统一体验规范**
   - 尤其是危险操作确认、统一错误提示、表格页一致性、AI 结果可解释性
