# 系统整体路线图、容量估算与防跑偏约束

## 文档目标

本文件不是单纯的“容量公式说明”，而是 DMSX 后续演进时的**总方向约束**。  
它要回答 4 个问题：

1. 这个系统最终要演进成什么样。
2. 当前版本真实处于哪个阶段。
3. 下一阶段该做什么、不该做什么。
4. 如何判断系统是否开始偏航。

---

## 北极星目标

DMSX 的目标不是“做一个功能拼盘”，而是演进为一套具备以下特征的**企业级统一设备管控平台**：

- 控制面稳定：租户、设备、策略、命令、制品、合规、审计、AI 能力边界清晰。
- 数据面可扩展：设备长连接、命令投递、回执、遥测入口能够独立水平扩容。
- 设备代理可跨平台：Windows / Linux / Android 接入模型尽量统一，平台差异封装到边界层。
- 操作链路连续：从设备列表到详情、远控、远程桌面、结果查看必须连贯、可恢复、可审计。
- 架构可演进：允许从 workspace 多 crate 平滑演进到多服务，而不是反复推翻重写。

---

## 当前真实状态

当前系统已经完成的主路径是：

- `dmsx-api` 作为控制面主入口，负责 REST API、桌面会话、LiveKit token 签发、审计与编排。
- `dmsx-agent` 当前通过 **HTTP 控制面接口** 实现注册、心跳、命令轮询、遥测上报。
- 远程桌面主链路已切换为 **LiveKit WebRTC + Data Channel**。
- `dmsx-device-gw` 已有 proto/gRPC/health 骨架，但**尚未成为默认生产数据面主链路**。

结论：

- 当前阶段是“**控制面可用 + Agent 主链路闭环 + 数据面骨架预埋**”。
- 后续演进必须围绕“**从可用单体控制面，逐步收敛到稳定控制面 + 独立数据面**”推进。

---

## 不可偏离的演进原则

### 1. 先收敛主链路，再扩展功能面

优先级永远是：

1. 设备接入稳定
2. 命令闭环稳定
3. 远程桌面稳定
4. 审计与可观测稳定
5. 在此基础上再扩展 AI、网络、工作流等高级能力

禁止反向优先级：

- 为了做“炫”的 AI 能力，牺牲命令与设备主链路稳定性
- 为了做更多页面，放弃现有交互连续性和状态一致性

### 2. 控制面和数据面不能长期“双主通路”

在 `dmsx-device-gw` 真正成熟前：

- `dmsx-agent -> dmsx-api` 的 REST 通路视为**唯一正式主链路**
- `dmsx-device-gw` 只作为下一阶段的数据面演进目标

当切换到数据面长连接时，必须满足：

- enrollment 可用
- streaming 命令链路可用
- 离线重放可用
- 回执与审计闭环可用

在满足这些条件前，不允许把两套接入模型都宣传为“生产主路径”。

### 3. 所有新增能力都必须挂到既有领域

新增功能时必须先回答：

- 属于 `device` / `policy` / `command` / `artifact` / `compliance` / `desktop` / `ai` / `network` 中哪个领域？
- 是控制面能力、数据面能力，还是设备代理能力？
- 状态归属在 Postgres、ClickHouse、对象存储还是消息总线？

如果回答不清，说明设计还没成熟，不应直接开工。

### 4. 文档必须跟随代码演进

以下文档是“真相源”的一部分，改代码时必须同步：

- `README.md`
- `docs/ARCHITECTURE.md`
- `docs/API.md`
- `docs/FRONTEND.md`
- `docs/SCALING_AND_ROADMAP.md`
- `docs/CHECKLIST.md`

如果代码和文档长期不一致，系统会先在认知层面跑偏，再在实现层面跑偏。

## 能力演进总图

### A. 控制面

持续演进方向：

- 多租户与权限模型稳定
- 声明式策略与 revision/reconcile 模式成熟
- 命令状态机、幂等与审计语义稳定
- 桌面会话与设备详情操作连续性完善
- 前端围绕“列表 -> 详情 -> 操作 -> 结果”主链路持续优化

### B. 数据面

持续演进方向：

- 从“预留骨架”演进到“独立接入平面”
- 支持大规模设备长连接、背压、离线重放
- 命令/策略下发从轮询演进到 streaming
- 数据面只承接连接、投递、回执与流控，不演化成第二个控制面

### C. 设备代理

持续演进方向：

- 注册、遥测、命令、远控、远程桌面模块化
- 平台差异封装，不在业务流程中散落 `cfg!(target_os = ...)`
- 与控制面契约尽量结构化，减少长期依赖裸 JSON

### D. 平台高级能力

这部分必须建立在 A/B/C 稳定之上：

- AI 智慧管控
- 网络编排
- 工作流审批
- SBOM / 软件供应链
- 多区域灾备

## 分阶段路线图

### Phase 0：控制面可用闭环

参考周期：2–4 周

目标：

- 租户、设备注册、心跳、设备影子、命令下发、执行结果、基础审计全部闭环
- Agent 通过 REST 通路稳定运行
- 前端完成列表/详情/基础操作主路径

退出标准：

- 设备注册与重连稳定
- 命令具备幂等、TTL、状态机
- 前端能无歧义查看命令执行结果
- 核心错误返回与日志字段可定位

### Phase 1：策略与控制面体验增强

参考周期：4–8 周

目标：

- 声明式策略、reconcile、分组标签、制品仓库、灰度发布
- 设备详情、远控历史、桌面状态机、URL 深链等体验增强
- 文档、DTO、前后端契约收口

退出标准：

- 策略 revision 机制稳定
- 设备详情操作链路连续
- 前端主要页面与文档结构一致
- 远程桌面主链路稳定运行

### Phase 2：数据面独立化

参考周期：8–12 周

目标：

- `dmsx-device-gw` 成为正式数据面接入层
- mTLS enrollment、命令/策略 streaming、离线重放、网关背压完成
- 控制面与数据面角色彻底分离

退出标准：

- Agent 可通过 gateway 建立稳定长连接
- 设备离线/重连后命令不丢失
- 网关具有租户级限流与观测能力
- 不再依赖 HTTP 轮询作为主要命令路径

### Phase 3：企业级扩展

持续演进

目标：

- 网络编排增强
- 工作流审批
- SBOM、证书轮换、供应链安全
- 多区域部署与灾备

退出标准：

- 每个能力都能明确挂到既有 bounded context
- 不引入“旁路控制面”或“临时状态孤岛”

## 季度任务树（建议执行视角）

这一节不是替代上面的 Phase，而是把 Phase 映射成更容易执行的季度节奏。
原则上，季度目标优先服务“阶段退出标准”，而不是单独追求功能数量。

### Q1：把现在这套主链路做稳

必须完成：

- 收口控制面核心领域：设备、命令、影子、桌面、审计
- 明确 API 层、repo 层、query row、DTO、前端 view model 的边界
- 修复主链路中的断链、状态不同步、导航跳转不连续问题
- 完成远程桌面、远控、命令结果的基本稳定性验证
- 清理文档与实现不一致项

可以并行推进：

- `dto.rs` 继续拆分，避免继续膨胀
- 将大组件、大 handler 拆为更清晰的 service / hook / view 层
- 为高频主链路补少量高价值测试

本季度不建议重押：

- 不要提前把主要精力投入 AI 编排、复杂审批、多区域部署
- 不要在 gateway 尚未成型时同时维护两套正式数据面主路径

季度验收：

- 新成员按文档可独立拉起环境并完成一次设备接入和远控演示
- 代码结构比当前更清晰，而不是继续堆到少数超大文件中
- 前端主操作链路刷新、返回、分享链接后上下文不丢失

### Q2：把数据面从“预留”变成“可用”

必须完成：

- 为 `dmsx-device-gw` 明确最小正式职责：身份、连接、投递、回执、背压
- 完成设备身份认证与 enrollment 基础闭环
- 将命令/策略主路径逐步迁移到 streaming
- 设计离线重放、投递确认、失败重试语义
- 增加 gateway 观测指标与租户级限流策略

可以并行推进：

- 统一 Agent 与 gateway 的结构化协议
- 减少设备侧裸 JSON 拼装，逐步向明确协议对象收口
- 为大规模连接准备压测脚本和容量基线

本季度不建议重押：

- 不要在 gateway 只完成连接功能时就把策略/审批/AI 逻辑塞进去
- 不要让数据面承担租户配置、业务编排、复杂查询等控制面职责

季度验收：

- gateway 可以作为正式接入层承接真实设备连接
- 设备离线/重连后命令语义正确，系统不会靠人工补偿维持正确性
- 主要命令下发不再默认依赖 HTTP 轮询

### Q3：把平台做成“能运营”的系统

必须完成：

- 完善审计、告警、回放、失败定位、容量监控
- 建立批量操作、灰度发布、回滚护栏
- 补齐多租户隔离、配额、资源治理规则
- 让安全设计与部署手册能支撑准生产环境

可以并行推进：

- 接入外部通知、对象存储、安全系统
- 打通更完整的操作留痕和变更审批记录

本季度不建议重押：

- 不要为了追求平台“看起来完整”而引入大量旁路管理入口
- 不要让观测、告警、审计分散在多个互不一致的实现里

季度验收：

- 运维人员可以通过日志、指标、审计快速回答“谁在何时对哪台设备做了什么”
- 批量变更具备最基本的保护、回滚和告警机制

### Q4：再上企业级增量能力

必须完成：

- 将 AI、审批、网络编排、SBOM 等能力明确挂靠到既有领域边界
- 为多区域、容灾、跨环境部署形成清晰策略
- 明确对外集成接口与平台治理模型

可以并行推进：

- 自然语言助手、异常检测、推荐能力试点
- 对接 IAM / SIEM / EDR / 工单系统

本季度不建议重押：

- 不要把试点能力直接做成平台级硬依赖
- 不要为了展示“智能化”而弱化已有主链路的确定性和可审计性

季度验收：

- 新增企业能力不会破坏原有控制面与数据面的清晰边界
- 平台已经具备从单系统走向企业运维底座的基本治理能力

## 按领域演进地图（防止任务混线）

这一节用于回答一个常见问题：
“同一个季度里，哪些事情可以并行做，哪些事情必须先后推进？”

建议后续把任务拆分到以下泳道中，再决定优先级与依赖关系。

### 1. 设备接入域

领域目标：

- 让设备注册、身份识别、在线状态、重连恢复形成稳定闭环

当前重点：

- 注册、心跳、设备影子、最后在线时间、基础接入稳定性

后续演进：

- enrollment、设备身份、长连接接入、离线恢复

与其他领域的依赖：

- 是命令域、桌面域、策略域的前置基础
- 不应承载复杂业务编排和审批逻辑

### 2. 命令执行域

领域目标：

- 让命令从创建、投递、执行、回执到审计都具备一致语义

当前重点：

- 命令状态机、TTL、幂等键、结果查询、执行中自动刷新

后续演进：

- streaming 下发、离线重放、批量命令、回滚与灰度语义

与其他领域的依赖：

- 依赖设备接入域保证设备可达
- 是远控、策略、批量变更的底层执行基础

### 3. 远控与桌面域

领域目标：

- 让“进入设备 -> 发起操作 -> 获取反馈 -> 安全退出”连续可用

当前重点：

- 远控历史分页、桌面会话状态、连接反馈、断开语义、URL 上下文恢复

后续演进：

- 更稳定的媒体链路、操作留痕、权限控制、审批挂接、会话回放

与其他领域的依赖：

- 依赖命令域和设备接入域
- 其体验质量会直接影响用户对整个平台“是否好用”的判断

### 4. 策略与编排域

领域目标：

- 让平台从“单条命令控制”演进到“声明式、可回滚、可审计”的持续管控

当前重点：

- revision / reconcile、标签分组、策略发布主语义

后续演进：

- 灰度、分批、生效窗口、失败回退、审批工作流挂接

与其他领域的依赖：

- 依赖命令域稳定后才能真正做强
- 不应绕过命令域和审计域另造状态机

### 5. 数据面网关域

领域目标：

- 让设备海量接入、投递、回执、流控成为独立能力，而不是 API 的附属逻辑

当前重点：

- 明确 `dmsx-device-gw` 的最小正式职责和边界

后续演进：

- mTLS、streaming、背压、租户配额、lag 观测、故障退化

与其他领域的依赖：

- 是设备接入域和命令域在规模化阶段的基础设施承载层
- 不应演化成第二控制面

### 6. Agent 平台能力域

领域目标：

- 让 Agent 的平台差异、能力模块、协议适配都能持续演进而不把主流程拖乱

当前重点：

- 注册、遥测、命令执行、远控、桌面模块化

后续演进：

- 平台抽象层、结构化协议、能力探测、离线缓存、本地审计

与其他领域的依赖：

- 同时连接控制面和数据面，是边界最容易被侵蚀的区域
- 必须避免把大量业务判断堆进 Agent

### 7. 前端体验域

领域目标：

- 让“列表 -> 详情 -> 操作 -> 结果”成为稳定、可分享、可回退的连续操作链

当前重点：

- URL 状态同步、详情抽屉上下文保持、状态刷新一致、错误反馈清晰

后续演进：

- 批量操作体验、审批交互、回放视图、跨页面状态一致性

与其他领域的依赖：

- 强依赖控制面契约稳定
- 是最早暴露后端边界混乱问题的地方

### 8. 安全与运营域

领域目标：

- 让系统不仅“能跑”，还“敢上线、可追责、可恢复”

当前重点：

- 审计、日志、错误定位、基本权限边界、部署手册一致性

后续演进：

- 证书轮换、SBOM、供应链安全、告警值班、灾备演练

与其他领域的依赖：

- 横切所有领域，但不应以“安全例外”方式破坏主链路一致性

### 9. AI 与智能管控域

领域目标：

- 在不破坏确定性与可审计性的前提下增强运营效率

当前重点：

- 保持设计抽象清晰，先不要反向侵入主链路

后续演进：

- 异常检测、推荐、自然语言助手、知识上下文、策略建议

与其他领域的依赖：

- 必须建立在命令、审计、策略、权限稳定之后
- 默认是增强层，不应成为控制面主路径硬依赖

### 领域间的主依赖顺序

推荐按下面的主顺序理解依赖，而不是把所有事情并列推进：

1. 设备接入域稳定
2. 命令执行域稳定
3. 远控与桌面域、前端体验域一起补连续性
4. 策略与编排域做强
5. 数据面网关域独立承接规模化需求
6. 安全与运营域补齐治理能力
7. AI 与智能管控域作为增强层逐步接入

### 领域拆任务时的约束

后续若按领域拆 `CHECKLIST`，建议遵循：

1. 一个任务只能有一个主领域归属，避免多头牵引。
2. 横切任务必须写清“主领域 / 依赖领域 / 验收口径”。
3. 如果某项需求无法判断归属，优先怀疑边界设计不清，而不是直接开工。
4. 任何新能力都不能绕过命令、审计、权限三条主约束单独落地。

---

## 决策顺序与取舍规则

后续做技术决策时，按以下顺序取舍：

1. 是否提升主链路正确性和稳定性？
2. 是否让控制面 / 数据面 / Agent 的边界更清楚？
3. 是否让状态归属更单一、更可审计？
4. 是否减少长期维护成本？
5. 是否改善用户体验？
6. 最后才是“是否更先进”或“是否更酷”

如果某个方案只是“技术更先进”，但让边界更乱、维护更难、文档更难同步，就不应该优先采用。

## 防跑偏信号

出现以下信号，说明系统正在偏航：

### 架构偏航

- `dmsx-api` 和 `dmsx-device-gw` 同时承担控制面职责
- Agent 同时维护两套长期并存的正式接入逻辑
- 新能力无法明确归属到既有领域

### 代码偏航

- `main.rs`、`dto.rs`、`handlers.rs` 持续膨胀，不拆层
- DTO、Repo Row、领域模型混在一起
- 组件中直接写复杂 API 协调与业务判断

### 产品偏航

- 新页面越来越多，但主操作链路越来越断裂
- 远控、桌面、命令详情之间状态不一致
- 用户需要靠“记忆流程”而不是系统反馈来完成操作

### 文档偏航

- 架构文档描述的主链路和实际代码不一致
- README 仍写旧能力，代码已经变了
- Checklist 不再反映真实完成度

## 容量估算方法

记：

- \(N\)：在线设备数
- \(f_h\)：心跳频率（次/秒/设备），例如 1/60
- \(f_c\)：人均命令下发速率（条/秒）
- \(S_r\)：平均回执大小（字节）

**心跳写入（ClickHouse）**

\[
Q_{heartbeat} \approx N \times f_h
\]

**命令与回执（消息总线 + CH）**

\[
Q_{cmd} \approx \lambda_{commands\_in} + N \times p_{exec}
\]

其中 \(p_{exec}\) 为设备侧执行完成产生回执的速率。JetStream / Kafka 分区数按峰值 \(Q_{cmd}\) 与单分区上限（经验 5–15MB/s）估算。

**网关连接**

- 每连接内存：order(数十 KB)（含 buffer）；\(N\) 百万级时需 **多副本 + 四层负载均衡 + 连接粘性（可选）**。

**Postgres**

- 热路径：设备行更新（`last_seen_at`）易成热点 → 使用 **批量合并写入**（秒级聚合）或 **先写 Redis 再刷 PG**。

## 背压与离线

| 场景 | 策略 |
|------|------|
| 网关过载 | 连接限流、租户级 quota、gRPC `RESOURCE_EXHAUSTED` |
| 消息堆积 | JetStream 保留策略 + 消费者 lag 告警；命令 TTL 到期标记 `expired` |
| 设备离线 | Agent 磁盘队列；上线后按 `cursor` 重放 `StreamCommands` |
| 大文件证据 | 分块上传 `UploadEvidence`；直传 S3 multipart |

说明：

- 当前“网关过载”“离线重放”更多对应 **Phase 2 目标形态**
- 当前主链路阶段仍应优先保障 REST Agent 通路的正确性和简洁性

---

## 幂等与一致性

- **命令**：部分唯一索引 `UNIQUE (tenant_id, idempotency_key) WHERE idempotency_key IS NOT NULL`。仅当提供幂等键时约束唯一，无键命令可重复创建。
- **策略**：`policy_revisions` 只追加；设备侧记录 `last_policy_revision_id`。
- **回执**：`command_id + device_id + attempt` 去重（CH ReplacingMergeTree 或 PG UPSERT）。
- **桌面会话**：以 `session_id` 为唯一生命周期主键，不再以 `device_id` 级别覆盖会话。

---

## 风险与缓解

- **热点行**：心跳合并、异步刷盘、CH 承担明细。
- **租户噪声**：租户级隔离（连接、Topic、配额）。
- **策略错误**：灰度 + 自动回滚（`rollout` 元数据）+ 只读审计。
- **双主链路漂移**：在 gateway 完整可用前，明确 REST Agent 是唯一正式主链路。
- **文档滞后**：每次能力迁移同时更新 Architecture / Frontend / README / Checklist / Roadmap。

## 未来 1 / 2 个迭代优先级清单

本节用于把 roadmap 继续下钻成“最近就能执行”的 backlog。
这里默认一个迭代以 1-2 周为参考，不强调绝对时长，强调优先级和验收顺序。

### Iteration 1：先收口主链路稳定性与结构边界

本迭代目标：

- 让当前控制面、Agent、前端主链路达到“连续可演示、连续可排障”的状态
- 继续消化当前代码结构中的膨胀点，避免 roadmap 还没落地，代码先失控

P0：

- 修复当前阻塞编译或验证的问题，优先清掉 `dmsx-api` 的已知构建阻塞项
- 收口 `dmsx-api` 中 DTO / query row / repo / service 的边界，避免继续混写
- 继续稳定设备详情、命令详情、远控、桌面之间的状态一致性
- 确认 LiveKit 远程桌面主链路的创建、连接、断开、重连、回收语义稳定
- 校正文档与实现差异，确保 README 与架构文档能支撑一次完整演示

P1：

- 拆分继续膨胀的文件，优先处理 `dto.rs`、大 handler、大前端组件
- 为命令状态机、桌面会话生命周期补充少量高价值验证
- 梳理错误返回与日志字段，保证排障时能串起一次完整操作链

本迭代完成判据：

- 主链路可以从设备接入一直走到命令执行、远控或桌面演示，中间无明显断链
- 当前阻塞构建问题被清理，至少核心 crate 能通过基础检查
- 文档描述与代码现状基本一致，不再出现“看文档无法拉起或无法理解现状”
- 主要模块没有继续朝更大的单文件、单函数恶化

本迭代默认不做：

- 不提前扩张审批、AI、多区域、复杂安全平台能力
- 不在 gateway 未成型前把主要精力转去做正式数据面切换

### Iteration 2：为数据面独立和策略演进做准备

本迭代目标：

- 在不破坏当前 REST 主链路稳定性的前提下，开始铺设 gateway 和策略域的下一步基础
- 把“可扩展”需要的协议、边界、观测准备起来，而不是直接堆功能

P0：

- 明确 `dmsx-device-gw` 的最小正式职责与不做事项
- 设计并收口 Agent 与 gateway 的结构化协议，减少裸 JSON 漫游
- 为命令 streaming / 离线重放 / 投递确认定义最小语义
- 为设备身份、enrollment、连接观测准备最小接口和数据结构

P1：

- 为策略 revision / reconcile 补一版更稳定的领域模型和接口边界
- 预留 gateway 的限流、lag、失败告警指标口径
- 为批量操作、灰度发布、回滚护栏补设计约束，不急于全部实现

本迭代完成判据：

- `dmsx-device-gw` 不再只是“未来会做”，而是有清晰边界、协议方向、演进入口
- Agent 和控制面之间的契约比现在更结构化，而不是继续堆叠临时字段
- 命令从当前模型向 streaming 演进的路径已经明确，不会后续推翻重来
- 策略域开始形成独立语义，但没有绕开命令与审计域单独生长

本迭代默认不做：

- 不急于把 gateway 做成全功能控制面替代物
- 不急于把 AI 助手或推荐能力接到主操作链路上

### 最近两轮迭代的总顺序

如果资源有限，建议按下面顺序推进，而不是平均用力：

1. 先清主链路阻塞项和结构边界问题
2. 再补主链路验证、日志、错误定位、文档一致性
3. 然后为 gateway / streaming / enrollment 铺协议和边界
4. 最后再进入策略增强和更远期的平台能力

## 里程碑执行规则

为了避免 roadmap 只停留在文档层，后续执行必须遵循：

1. 每个阶段开始前，先把目标拆成 `CHECKLIST` 和具体任务，不直接跳进实现。
2. 每个阶段结束前，必须回看本文件中的“完成判据”，而不是只看代码是否写完。
3. 如果新增需求不属于当前阶段主目标，默认延后，除非它能显著提升主链路稳定性。
4. 如果发现架构、代码、产品或文档偏航信号，必须优先纠偏，而不是继续叠加功能。
5. 阶段推进顺序允许调整节奏，但不允许打乱“控制面稳定 -> 数据面独立 -> 企业级扩展”的主序。
